# Roadmap — POS + Akuntansi

## Tujuan

Membangun sistem POS + akuntansi dengan:

- **Mobile POS offline-first** (Flutter) untuk kasir di outlet.
- **Web app** (Laravel 13 + Inertia + React) untuk stok, gudang, dan akuntansi.
- **API Sanctum** sebagai jembatan antara mobile dan backend.
- **Database**: MySQL.
- **Costing**: Moving Average — bukan FIFO/LIFO.
- **Akuntansi**: double-entry, PPN masuk/keluar tercatat sebagai jurnal, bukan sekadar field pajak.
- **Skema**: satu outlet untuk sekarang, tapi struktur data multi-outlet-ready sejak awal (kolom `outlet_id`/`warehouse_id` sudah ada di semua tabel transaksi) — supaya ekspansi ke outlet lain tidak butuh migrasi struktural besar.

## Status: SELESAI (test hijau di MySQL)

- **Skema database** — 23 tabel inti (outlets, warehouses, uoms, uom_conversions, item_categories, items, product_categories, products, product_components, suppliers, purchase_orders, purchase_order_lines, goods_receipts, goods_receipt_lines, stock_movements, stock_opnames, stock_opname_lines, sales, sale_lines, tax_rates, accounts, journals, journal_lines) + model Eloquent dengan relasi lengkap + seeder fondasi (outlet, warehouse, UOM, Chart of Accounts, PPN 11%).
- **`InventoryService`** — Moving Average: `recordInbound()`/`recordOutbound()`, locking (`lockForUpdate`) untuk konsistensi `running_qty`/`running_average_cost`, item `cost_only` (mis. air) tidak dilacak stok.
- **`PostingService`** — posting jurnal double-entry, validasi SUM(debit) = SUM(credit) sebelum commit (`UnbalancedJournalException` kalau tidak seimbang), auto-generate nomor jurnal, resolve akun by kode (bukan ID hardcode).
- **`SaleService`** — alur penjualan end-to-end: sale + sale_lines → baca BOM (`product_components`) → potong stok tiap komponen via `InventoryService` (dengan konversi UOM resep → base_uom item) → akumulasi HPP → posting jurnal penjualan (Kas/Penjualan/PPN Keluaran/HPP/Persediaan).
- **`PurchaseService`** — alur pembelian + penerimaan: PO hanya mencatat niat beli (tidak menyentuh stok/jurnal); `receiveGoods()` mengonversi qty dari `purchase_uom` ke `base_uom`, memanggil `InventoryService::recordInbound()`, posting jurnal pembelian (Persediaan/PPN Masukan/Hutang Usaha), dan menghitung status PO (`open`/`partial`/`received`) **per baris PO** lewat kolom `goods_receipt_lines.purchase_order_line_id` — jadi dua baris PO untuk item yang sama tetap dilacak independen (bukan lagi diagregasi per item).
- **`StockOpnameService`** — hitung fisik stok dua fase: `startOpname()` men-snapshot `system_qty` (item `cost_only` ditolak karena stoknya memang tidak dilacak) tanpa menyentuh stok/jurnal; `postOpname()` menghitung selisih terhadap stok LIVE saat posting (bukan snapshot awal, supaya `running_qty` akhir dijamin sama persis dengan hasil hitung fisik walau ada transaksi lain di tengah proses), surplus dicatat via `recordInbound()` di `running_average_cost` saat ini (average tidak bergeser), shrinkage via `recordOutbound()`, lalu posting satu jurnal penyesuaian yang meringkas total surplus & shrinkage ke akun "Selisih Persediaan" (`5-2000`) — surplus di-credit, shrinkage di-debit (satu akun untuk dua arah, keputusan disengaja: lihat "Utang teknis").
- **UI web — CRUD master data (sebagian)**: UOM, Supplier, Item, dan Product (+ editor BOM/`product_components` dinamis) — controller resource (`app/Http/Controllers/Master/*`), halaman Inertia+React (`resources/js/Pages/Master/*`), nav link di `AuthenticatedLayout`. Diverifikasi end-to-end lewat browser (Playwright): login → create/edit/delete tiap entity → BOM tersimpan & ter-*load* ulang dengan benar → nol console error. **Sengaja di-skip untuk sekarang**: CRUD Item Category & Product Category (field-nya nullable di form, bisa ditambah belakangan), dan CRUD Chart of Accounts/Tax Rate (sudah ter-seed lewat `FoundationSeeder`, terlalu sensitif untuk diedit bebas dari UI tanpa kontrol lebih — lihat "Utang teknis").
- **UI web — Layar kasir** (`/kasir`, `Kasir\SaleController`, `resources/js/Pages/Kasir/Index.jsx`): grid produk aktif (cari by nama) + keranjang di client, checkout memanggil `SaleService::createSale()` langsung (`payment_method` selalu `cash` untuk sekarang). Total di keranjang cuma estimasi tampilan di client — server selalu hitung ulang dengan bcmath lewat `SaleService`, jadi tidak ada nilai uang yang dipercaya dari client. Kegagalan (mis. `RuntimeException` dari BOM/UOM) ditangkap di controller, di-`report()` untuk log, lalu jadi flash message ramah tanpa 500. Diverifikasi end-to-end lewat browser sungguhan: checkout 2x produk dengan BOM asli → stok komponen berkurang benar, jurnal otomatis seimbang (Kas/Penjualan/PPN Keluaran/HPP/Persediaan) dengan angka yang cocok dengan preview di client.
- **UI web — Layar transaksi pembelian** (`/pembelian/purchase-orders`, `Pembelian\PurchaseOrderController` + `Pembelian\GoodsReceiptController`): buat PO dengan baris dinamis (item/qty/purchase_uom/harga/pajak, auto-isi UOM dari default item), halaman detail PO menampilkan status, baris (dipesan vs sudah diterima — dua satuan berbeda ditampilkan eksplisit dengan kode UOM masing-masing biar tidak ambigu), dan riwayat penerimaan; form "Terima Barang" per baris PO memanggil `PurchaseService::receiveGoods()` langsung. **Bug kecil ditemukan & diperbaiki sambil membangun ini**: `PurchaseService::receiveGoods()` sebelumnya tidak memvalidasi bahwa `purchase_order_line_id` yang dikirim benar-benar milik PO yang diproses — celah ini baru terlihat begitu ID baris datang dari input pengguna (form web), bukan dari test yang selalu memakai ID yang benar. Sudah di-scope dengan `where('purchase_order_id', $po->id)` + validasi `Rule::exists(...)->where(...)` di controller + test regresi. Diverifikasi end-to-end lewat browser: buat PO 100 pcs @15000 +PPN → terima 60 (status `partial`) → terima 40 sisanya (status `received`) → 2 jurnal terpisah (satu per goods receipt) masing-masing seimbang, stok akhir 100 dengan average cost benar.
- **UI web — Layar stock opname** (`/stock-opname`, `StockOpnameController`): dua fase sesuai desain `StockOpnameService` — `Create.jsx` memilih gudang/tanggal/item (item picker cuma menampilkan item `costing_type=stocked`, item `cost_only` di-filter di query controller supaya tidak memicu error yang sudah pasti terjadi di service), `startOpname()` bikin sesi draft; `Show.jsx` menampilkan `system_qty` per baris dan input hasil hitung fisik (di-prefill dengan `system_qty` sebagai default masuk akal) selama status masih `draft`, lalu "Posting Hasil Opname" memanggil `postOpname()` langsung. Setelah `completed`, halaman yang sama berubah jadi read-only menampilkan `diff_qty` (hijau untuk surplus, merah untuk shrinkage). Diverifikasi end-to-end lewat browser: opname 2 item sekaligus (satu surplus +10, satu shrinkage -10) → stok & average cost masing-masing benar, satu jurnal dengan 4 baris seimbang (Dr Persediaan 10000 + Dr Selisih 20000 = Cr Selisih 10000 + Cr Persediaan 20000).
- **API Sanctum untuk mobile** (`routes/api.php`, prefix `/api/v1`): `POST /login` (issue token), `POST /logout` (revoke token via `currentAccessToken()->delete()`), `GET /products` (katalog aktif untuk cache offline), `POST /sales` (sync penjualan dari HP kasir, panggil `SaleService::createSale()`). **Perubahan desain penting pada `SaleService`**: `createSale()` sekarang menerima `local_uuid` opsional dari pemanggil dan bersifat idempotent — kalau `local_uuid` itu sudah pernah diproses, method langsung mengembalikan `Sale` yang sudah ada tanpa memotong stok atau posting jurnal lagi (termasuk penanganan race condition lewat unique constraint violation catch, bukan cuma pre-check biasa). Ini krusial untuk offline-first: HP kasir generate `local_uuid` saat transaksi terjadi offline, dan retry submit yang sama kalau koneksi putus harus aman. Diverifikasi lewat test (retry local_uuid sama → sale & stok tidak dobel) **dan** lewat `curl` sungguhan ke server yang jalan (bukan cuma PHPUnit in-process): login → 401 tanpa token → sukses dengan token → POST sale dua kali dengan `local_uuid` sama → response identik (id & timestamp sama), stok cuma terpotong sekali. Endpoint pembelian/stock-opname belum ada di API (mobile POS untuk sekarang cuma butuh alur jual, bukan pembelian/opname — lihat "Utang teknis").
- **Laporan keuangan** (`/laporan/neraca`, `/laporan/laba-rugi`, `FinancialReportController` + `FinancialReportService`): Neraca (as-of tanggal tertentu) dan Laba Rugi (rentang tanggal), keduanya menjumlah `journal_lines` per akun langsung dengan bcmath (bukan kolom saldo yang di-cache). **Keputusan desain penting (dikonfirmasi ke user)**: CoA belum punya akun Ekuitas/Modal dan sistem ini tidak melakukan jurnal penutup periode — supaya Neraca tetap balance, laba/rugi berjalan (Pendapatan − Beban akumulatif sampai tanggal Neraca) dihitung live dan ditampilkan sebagai baris "Laba/Rugi Berjalan (belum ditutup)" di bagian Ekuitas — ini murni konstruksi tampilan laporan, bukan jurnal sungguhan. Diverifikasi test (termasuk filter tanggal: jurnal setelah as-of/di luar rentang tidak ikut terhitung) **dan** lewat browser dengan transaksi pembelian+penjualan nyata — angka di layar cocok persis dengan perhitungan manual (Aset 156.500 = Liabilitas 116.500 + Ekuitas 40.000).
- **Concurrency test untuk `lockForUpdate()`** (`tests/Feature/Concurrency/InventoryServiceConcurrencyTest`, `PostingServiceConcurrencyTest`) — membuktikan lock di `InventoryService`/`PostingService` benar-benar menyerialisasi dua transaksi MySQL yang **sungguhan paralel** (dua proses OS terpisah lewat `Illuminate\Support\Facades\Process`, bukan dua panggilan berurutan dalam satu proses PHP — RefreshDatabase sengaja TIDAK dipakai di test ini karena akan membungkus semuanya dalam satu transaksi bersama, membuat lock tidak pernah bentrok). Proses "pemegang lock" adalah command Artisan tersembunyi (`concurrency-test:hold-inventory-lock`/`hold-journal-lock`, hanya jalan di `APP_ENV=testing`) yang menahan row lock sungguhan beberapa detik sebelum commit; proses test utama mengukur bahwa panggilannya sendiri benar-benar ter-*block* selama itu (bukti waktu), lalu memverifikasi hasil akhir benar seolah dieksekusi berurutan — bukan sekadar "tidak error": `running_qty`/`running_average_cost` gabungan yang tepat (150/1333.3333, bukan salah satu menimpa yang lain) untuk `InventoryService`, dan nomor jurnal tidak pernah duplikat (termasuk skenario "jurnal pertama di bulan yang benar-benar kosong") untuk `PostingService`. Diverifikasi tidak flaky lewat 3x run berturut-turut.
- **Perbaikan akar race condition penomoran jurnal** (ditemukan & diperbaiki *sebelum* menulis test di atas, karena test yang benar akan mengekspos bug produksi nyata): `PostingService::generateNumber()` sebelumnya mengandalkan `SELECT ... FOR UPDATE` pada baris `journals` yang mungkin belum ada sama sekali (bulan kosong) — gap-lock untuk klausa `LIKE` tidak cukup terjamin untuk dipercaya pada buku besar. Diganti dengan tabel `journal_number_sequences` (satu baris per periode `Ym`, di-upsert atomik dulu supaya SELALU ada baris untuk dikunci sebelum `lockForUpdate()`, baru increment). Tiga lapis: (1) counter table = mekanisme utama yang menghapus ketergantungan pada gap-lock, (2) unique constraint pada `journals.number` (sudah ada) = jaring pengaman mutlak di level database, (3) retry loop di `post()` = kalau tetap terjadi duplicate-key (SQLSTATE 23000 + error 1062 + spesifik index `journals_number_unique`, bukan `QueryException` apa pun), coba lagi dengan **transaksi baru yang segar** (bukan retry di dalam transaksi yang sama — itu tidak akan pernah melihat commit pesaing). Diverifikasi lewat test sekuensial (dua post di bulan kosong → 0001 lalu 0002; baris counter ter-rollback kalau `post()` gagal karena sebab lain) sebelum lanjut ke concurrency test di atas.
- **PPN tax-inclusive + saklar global** (2026-07-10, rancangan dikonfirmasi user sebelum implementasi — lihat gerbang konfirmasi di `docs/PRINCIPLES.md`): `products.sell_price` sekarang berarti harga SUDAH TERMASUK PPN (bukan lagi harga+pajak ditambahkan). Tabel baru `company_settings` (singleton, `id=1`, kolom `ppn_active`, di-seed lewat `FoundationSeeder`) — saklar tingkat sistem untuk kondisi PKP vs non-PKP, dibaca `SaleService::createSale()` sehingga otomatis konsisten antara layar kasir web dan API mobile (keduanya panggil service yang sama). Per baris: `line_net = line_inclusive ÷ (1 + tax_rate)` (bcdiv truncate skala 4), `line_tax = line_inclusive − line_net` (bcsub, BUKAN perkalian independen `net × rate`) — pengurangan menjamin `net + tax` selalu eksak sama dengan `line_inclusive`, tanpa residu pembulatan ganda. `grand_total` dihitung langsung dari jumlah harga inclusive tiap baris (bukan dari `subtotal+tax_total`), lalu `SaleService` men-assert keduanya sama persis (`UnreconciledSaleTotalException` kalau tidak — jaring pengaman, seharusnya tidak pernah terpicu kalau rumus benar). Produk tanpa `tax_rate_id` atau saat `ppn_active=false` tidak kena pajak sama sekali (seluruh harga masuk sebagai penjualan bersih) — satu baris kode menangani ketiga kasus (saklar off / saklar on+kena pajak / saklar on+tidak kena pajak) tanpa percabangan terpisah, termasuk keranjang campuran (sebagian baris kena pajak, sebagian tidak, dalam satu sale yang sama). `PostingService` sendiri tidak berubah sama sekali — balance jurnal (`Dr Kas+HPP = Cr Penjualan+PPN Keluaran+Persediaan`) terjamin aljabar karena `grand_total ≡ subtotal+tax_total` by construction, bukan kebetulan. `GET /api/v1/products` menambahkan `meta.ppn_active` (di response yang sudah ada, bukan endpoint baru) supaya mobile bisa mengestimasi total struk offline sebelum sync. Layar kasir web (`Kasir/Index.jsx`) juga diperbarui ke rumus inclusive yang sama untuk preview keranjangnya. **Struktur jurnal PPN Keluaran sudah siap untuk laporan rekap per periode nanti** (setiap baris jurnal PPN Keluaran punya `journals.date` + `source_type/source_id` menunjuk balik ke `Sale` asal — laporan tinggal `SUM(journal_lines.credit) WHERE account=2-1100 AND date BETWEEN ...`, tidak perlu kolom baru) — laporannya sendiri belum dibangun, sengaja ditunda sampai mekanisme transaksi ini teruji. UI admin untuk toggle `ppn_active` SENGAJA ditunda (tinker/seeder dulu, konsisten dengan CoA/Tax Rate yang sudah lebih dulu ditunda dari UI) — lihat "Utang teknis". **Mobile (`pos_mobile`) sudah diperbarui ke model inclusive yang sama (Tahap 4a, 2026-07-10) — angka terverifikasi cocok persis di HP; peringatan pajak-dobel yang sebelumnya ada di sini sudah tidak berlaku.**

## Utang teknis

- **PHP `is_writable()` false-negative di `bootstrap/cache` pada environment Windows ini.** Direktori itu punya ACL Windows `Everyone:(DENY)(DC)` yang ter-*inherit* (bisa dicek lewat `icacls`), membuat `is_writable()` PHP mengembalikan `false` walau tulis file sungguhan (`file_put_contents`) berhasil. Ini membuat `Illuminate\Foundation\PackageManifest`/`ProviderRepository` melempar exception "directory must be present and writable" setiap kali Laravel perlu MENULIS ulang `bootstrap/cache/packages.php` atau `services.php` (dipicu oleh `composer update`/`install`, `php artisan package:discover`, atau `php artisan vendor:publish`) — **membaca** cache yang sudah ada tetap normal. Insiden ini sempat terjadi saat `php artisan install:api` dijalankan (menghapus cache lama lalu gagal menulis yang baru), membuat SEMUA perintah `artisan` gagal sementara; sudah dipulihkan dengan menulis ulang kedua file manifest secara manual (lewat `file_put_contents`, bukan lewat Laravel, karena is_writable) mereplikasi persis logika `PackageManifest::build()`/`Application::registerConfiguredProviders()`. **Implikasi ke depan**: hindari menjalankan `composer update/install`, `artisan package:discover`, atau `artisan vendor:publish` kalau memungkinkan; kalau terpaksa dan artisan tiba-tiba error "bootstrap/cache directory must be present and writable" setelahnya, itu gejala bug ini, bukan regresi kode — perbaikannya adalah menulis ulang manifest secara manual seperti di atas, BUKAN mengubah ACL Windows folder tersebut (di luar cakupan proyek, berpotensi ada alasan keamanan di baliknya).
- ~~Celah TOCTOU di `StockOpnameService::postOpname()` masih belum diuji di bawah race condition nyata~~ — **SELESAI (2026-07-11)**. `InventoryService` mendapat method baru `lockAndReadCurrentStock()` (kunci ledger item, lalu baca qty+avg cost FRESH dari dalam lock yang sama) — `postOpname()` sekarang memakainya, bukan `currentStock()`/`currentAverageCost()` tanpa lock seperti sebelumnya, menutup jendela antara baca-stok dan tulis-penyesuaian. Baris-baris opname juga diurutkan `item_id` ASC sebelum diproses (cegah deadlock lock-order lintas-item). Ditemukan sekaligus race terpisah: dua panggilan `postOpname()` untuk OPNAME YANG SAMA (double-click/retry) sama-sama lolos cek status `draft` sebelum salah satu commit — ditutup dengan `lockForUpdate()` pada baris `stock_opnames` di awal transaksi, re-check status di dalam lock. Dibuktikan dengan 2 concurrency test baru (`tests/Feature/Concurrency/StockOpnameServiceConcurrencyTest.php`, pola sama persis dengan `InventoryService`/`PostingService`: subprocess OS sungguhan, tanpa RefreshDatabase) — keduanya sengaja dibuktikan MERAH dulu terhadap kode sebelum fix (test 1: hasil basi 160 vs benar 130; test 2: proses kedua tidak terlempar exception) sebelum fix diterapkan dan menjadi hijau. Ditambah 1 unit test langsung (non-concurrency) yang membuktikan urutan `item_id` ASC benar-benar diterapkan.
- **Chart of Accounts hanya punya satu akun "Selisih Persediaan" (`5-2000`, expense/debit)** untuk surplus maupun shrinkage stock opname (keputusan disengaja, dikonfirmasi ke user). Surplus di-credit ke akun ini (mengurangi expense), shrinkage di-debit (menambah expense) — P&L akan menampilkan **net** selisih persediaan, bukan angka surplus dan shrinkage yang terpisah. Kalau nanti butuh laporan "nilai barang lebih" vs "nilai barang hilang" secara terpisah, perlu akun baru (mis. `4-2000` pendapatan lain-lain) + migrasi data historis.
- **Neraca tidak punya jurnal penutup periode** (lihat penjelasan di bagian SELESAI) — "Laba/Rugi Berjalan" akan terus tumbuh selamanya karena tidak pernah "ditutup" ke akun Modal di awal periode baru. Ini oke untuk laporan snapshot kapan saja, tapi kalau nanti butuh laporan "Laba Ditahan per tahun buku" yang benar-benar terpisah per tahun, perlu proses tutup buku beneran (jurnal penutup Revenue/Expense ke akun Modal/Laba Ditahan setiap akhir periode). "Aset Lancar" menaungi "Kas"+"Bank"), `FinancialReportService` perlu diupdate untuk rollup rekursif.
- **CRUD Chart of Accounts, Tax Rate, Item Category, dan Product Category belum ada di UI.** Keempatnya hanya bisa diubah lewat `FoundationSeeder`/tinker/migration untuk sekarang. Item Category dan Product Category memang nullable jadi tidak blocking; tapi kalau nanti perlu ubah PPN atau tambah akun baru dari CoA (mis. untuk memisah akun Selisih Persediaan di atas), belum ada jalur UI-nya.
- **Saklar `company_settings.ppn_active` belum ada UI admin** (keputusan disengaja, dikonfirmasi user 2026-07-10, konsisten dengan CoA/Tax Rate di atas) — untuk sekarang cuma bisa diubah lewat tinker (`App\Models\CompanySetting::current()->update(['ppn_active' => false])`) atau migrasi. Sesuai desain "PO hanya mencatat niat" ini rendah risiko selama belum diterima, tapi kalau salah input saat membuat PO, satu-satunya cara memperbaikinya sekarang adalah lewat tinker/DB langsung — belum ada alur pembatalan resmi (status `cancelled` sudah ada di enum tapi belum dipakai di mana pun).
